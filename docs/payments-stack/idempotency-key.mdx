---
title: "Idempotency Key"
---

An idempotency key is a unique client-generated value that Meld uses to recognize and de-duplicate retried requests. This page is for developers who want to safely retry payment requests — for example after a network timeout — without risking duplicate charges.

For example, if a request to create a charge does not respond due to a network error, you can retry the request with the same idempotency key to ensure no more than one charge is created.

<Warning>
**Reusing an idempotency key for a different request**

Idempotency keys must be unique per logical operation. **Reusing the same `Idempotency-Key` across different requests (different amount, different customer, different payment method) will cause Meld to return the original cached response instead of processing the new request.** This can silently mask new charges or refunds. Always generate a fresh UUID for every distinct operation, and only re-send the same key when intentionally retrying the exact same request.
</Warning>

## Sending an idempotent request

To send an idempotent request, provide an additional `X-Idempotency-Key` element in the request header. You can generate a random UUID v4 at [uuidgenerator.net](https://www.uuidgenerator.net).

Sample request using `X-Idempotency-Key`:

```bash
curl --location --request POST '{api_base_url}/payments/transactions' \
--header 'Authorization: BASIC {{Your API Key}}' \
--header 'X-Idempotency-Key: {{Idempotency Key}}' \
--header 'Content-Type: application/json' \
--data-raw '{
    "amount": "10",
    "currencyCode": "USD",
    "nonce": "meld-fake-valid-nonce",
    "vault": false,
    "serviceProvider": "AUTHORIZENET"
}'
```

<Tip>
Store the idempotency key alongside the original request in your system so a retry path can resend the exact same key.
</Tip>
